574  * The TQ3ComputeBounds flag passed to StartBoundingBox or StartBoundingSphere
575  * calls in the View. It's a hint to the system as to how it should
576  * compute the bbox of a shape:
577  *
578  * kQ3ComputeBoundsExact:
579  * Vertices of shapes are transformed into world space and
580  * the world space bounding box is computed from them. Slow!
581  *
582  * kQ3ComputeBoundsApproximate:
583  * A local space bounding box is computed from a shape's
584  * vertices. This bbox is then transformed into world space,
585  * and its bounding box is taken as the shape's approximate
586  * bbox. Fast but the bbox is larger than optimal.

1267 /*
1268  * QuickDraw 3D 1.5 Note:
1269  *
1270  * Shapes and Sets are now (sort of) polymorphic.
1271  *
1272  * You may call Q3Shape_Foo or Q3Set_Foo on a shape or a set.
1273  * The following calls are identical, in implementation:
1274  *
1275  * Q3Shape_GetElement = Q3Set_Get
1276  * Q3Shape_AddElement = Q3Set_Add
1277  * Q3Shape_ContainsElement = Q3Set_Contains
1278  * Q3Shape_GetNextElementType = Q3Set_GetNextElementType
1279  * Q3Shape_EmptyElements = Q3Set_Empty
1280  * Q3Shape_ClearElement = Q3Set_Clear
1281  *
1282  * All of these calls accept a shape or a set as their first parameter.
1283  *
1284  * The Q3Shape_GetSet and Q3ShapeSetSet calls are implemented via a new
1285  * element type kQ3ElementTypeSet. See the note above about
1286  * kQ3ElementTypeSet;
1287  *
1288  * It is important to note that the new Q3Shape_...Element... calls do not
1289  * create a set on a shape and then add the element to it. This data is
1290  * attached directly to the shape. Therefore, it is possible for an element
1291  * to exist on a shape without a set existing on it as well.
1292  *
1293  * In your application, if you attach an element to a shape like this:
1294  * (this isn't checking for errors for simplicity)
1295  *
1296  * set = Q3Set_New();
1297  * Q3Set_AddElement(set, kMyElemType, &data);
1298  * Q3Shape_SetSet(shape, set);
1299  *
1300  * You should retrieve it in the same manner:
1301  *
1302  * Q3Shape_GetSet(shape, &set);
1303  * if (Q3Set_Contains(set, kMyElemType) == kTrue) {
1304  * Q3Set_Get(set, kMyElemType, &data);
1305  * }
1306  *
1307  * Similarly, if you attach data to a shape with the new calls:
1308  *
1309  * Q3Shape_AddElement(shape, kMyElemType, &data);
1310  *
1311  * You should retrieve it in the same manner:
1312  *
1313  * if (Q3Shape_ContainsElement(set, kMyElemType) == kTrue) {
1314  * Q3Shape_GetElement(set, kMyElemType, &data);
1315  * }
1316  *
1317  * This really becomes an issue when dealing with version 1.0 and version 1.1
1318  * metafiles.
1319  *
1320  * When attempting to find a particular element on a shape, you should
1321  * first check with Q3Shape_GetNextElementType or Q3Shape_GetElement, then,
1322  * Q3Shape_GetSet(s, &set) (or Q3Shape_GetElement(s, kQ3ElementTypeSet, &set))
1323  * and then Q3Shape_GetElement(set, ...).
1324  *
1325  * In terms of implementation, Q3Shape_SetSet and Q3Shape_GetSet should only be
1326  * used for sets of information that are shared among multiple shapes.
1327  *
1328  * Q3Shape_AddElement, Q3Shape_GetElement, etc. calls should only be used
1329  * for elements that are unique for a particular shape.
1330  *
1331  */
1332 /*

1337  * CarbonLib: not available
